# Program Assign 設計書 v1.0

作成日：2026年7月22日  
対象リポジトリ：`Steveston147/program_assign`

## 1. 文書の目的

本書は、Program Assignの役割、対象範囲、画面、データ、運用方法、今後の拡張方針を定める設計基準です。

新しい機能を追加する際は、本書に定める役割から外れていないかを確認します。役割を変更する必要がある場合は、実装より先に本書を更新します。

## 2. アプリの目的

Program Assignは、短期留学生受入プログラムに関する職員の日別予定を共有する内部向けWebアプリです。

利用者が、次の情報を短時間で確認できることを目的とします。

- 誰が担当するか
- 何時から何時までか
- どのプログラムか
- 何を担当するか
- どこに集合し、どこへ行くか
- 役割、備考、確定状況

中心となる利用場面は、当日または近日の担当確認です。

## 3. 基本方針

### 3.1 閲覧を中心とする

Program Assignは、予定を見やすく共有するためのアプリです。

予定の正本はExcelとし、アプリ画面上で予定を直接作成・編集・削除しません。管理者はExcelを更新し、アプリへアップロードして反映します。

### 3.2 Program Managerとは分離する

Program AssignとProgram Managerは別アプリ、別リポジトリとして運用します。

- Program Assign：職員別の日別予定を確認する
- Program Manager：プログラム全体の進捗、期限、リスク、判断事項を管理する

両者を一つのアプリへ統合しません。将来のデータ連携は可能としますが、役割と画面は分離します。

### 3.3 小さな変更で保守する

開発は原則として、次の単位で進めます。

- 一つの論点につき一つのPR
- 必要最小限のファイルだけを変更
- 表示変更とロジック変更を可能な限り分離
- 大規模なリファクタリングを避ける
- Previewで確認してからMergeする

## 4. 利用者と権限

### 4.1 閲覧者

閲覧者は、ログイン後に次の操作を行えます。

- 日付を移動・選択する
- 職員名、プログラム、ステータスで絞り込む
- カード表示とガント表示を切り替える
- 予定の詳細を閲覧する
- 印刷する
- ログアウトする

### 4.2 管理者

管理者は、閲覧機能に加えて次の操作を行えます。

- Excelファイルをアップロードする
- 入力検証結果を確認する
- データ反映結果と最終更新日時を確認する

閲覧者用と管理者用の認証情報は分離します。

## 5. 画面構成

### 5.1 ログイン画面

共有パスワードを入力し、閲覧画面または管理者画面へ移動します。

### 5.2 閲覧画面

閲覧画面は、上から次の順に構成します。

1. ヘッダー
2. Excel読込状況と最終更新日時
3. 日付ナビゲーション
4. 職員名・プログラム・ステータスのフィルター
5. 対象日の見出し
6. カード・ガント表示切替
7. 予定表示
8. データ取扱いに関する補足

### 5.3 カード表示

社員ごとにカードを表示し、予定を開始時刻順に並べます。

予定カードでは、次の優先順位で情報を表示します。

1. 時間
2. ステータス
3. 予定名
4. プログラム名
5. 役割
6. 集合時間・集合場所
7. 行先
8. 備考

予定がない社員もカードを表示し、「本日の予定はありません」と示します。

### 5.4 ガント表示

職員別に、予定を時間軸上へ配置します。

- 表示時間：08:00〜18:00
- 予定の位置と幅は開始・終了時刻に対応させる
- 重複予定は別レーンに表示する
- プログラム別カラーを維持する
- 60分未満は開始時刻と予定名を優先表示する
- 60分以上は時間帯、プログラム名、予定名等を表示する
- 省略された内容はツールチップで確認できる

ガントは時間の長さを正確に示すことを優先し、文字表示のためにバー幅を実時間以上へ広げません。

### 5.5 印刷表示

カード表示を中心に、日別予定を紙またはPDFとして出力できる状態を維持します。

印刷時は、操作ボタンなど不要なUIを非表示にし、予定内容が欠けないことを優先します。

## 6. 日付とフィルター

### 6.1 日付

- 前日、翌日、今日へ移動できる
- 中央の日付欄から任意の日付を選択できる
- 今日の判定は日本時間で行う
- 表示対象日に予定がない場合も、空状態を明示する

### 6.2 フィルター

次の項目で絞り込みます。

- 職員名
- プログラム名
- ステータス

すべてのフィルターは同時に適用でき、「すべて」で解除できます。

## 7. データ設計

### 7.1 正本

予定データの正本はExcelです。

必須シート：

- `App_Data`
- `Master_Staff`

### 7.2 App_Data

必須列：

- `Date`
- `StaffName`
- `StartTime`
- `EndTime`
- `ProgramName`
- `GatheringPlace`
- `EventName`
- `Status`

任意列：

- `Role`
- `GatheringTime`
- `Destination`
- `Notes`
- `UpdatedAt`

シート名と英語ヘッダーは固定し、表示上の日本語名称とは分離します。

### 7.3 Master_Staff

職員名と表示順を管理します。

`DisplayOrder`を基準として、カード表示とガント表示の職員順を統一します。

### 7.4 読込優先順位

1. Vercel Blobへアップロード済みのデータ
2. `data/schedule.xlsx`
3. `data/schedule.seed.json`から生成したサンプルデータ

通常運用では、Vercel Blob上のアップロード済みデータを優先します。

## 8. 入力検証

Excelアップロード時に、少なくとも次を検証します。

- 必須シートの存在
- 必須列の存在
- 必須項目の未入力
- 日付形式
- 開始・終了時刻の形式
- 終了時刻が開始時刻より後であること
- 職員名など主要値の妥当性

エラーがある場合は保存せず、対象行と理由を管理者へ表示します。

## 9. セキュリティとデータ取扱い

- Excel本体を`public`配下へ置かない
- Excel本体をブラウザへ直接配信しない
- 認証済みユーザーへ必要なJSONだけを返す
- 未認証アクセスには401を返す
- APIレスポンスは`Cache-Control: no-store`とする
- アップロード済みデータはVercel Blobのprivate領域へ保存する
- 認証Cookieには署名を用いる
- 秘密情報は環境変数で管理する

## 10. 非機能要件

### 10.1 操作性

- 毎朝開いてすぐに予定を確認できる
- PCでは一覧性を重視する
- スマートフォンでも横にはみ出さず確認できる
- キーボード操作時にフォーカス位置が分かる

### 10.2 表示

- Slate系を基本に、選択状態はIndigo系で示す
- プログラム別カラーは予定の識別に使用する
- 装飾よりも可読性を優先する
- 長い文字列は崩さず、省略または折り返す

### 10.3 時刻

- 日付と今日判定は日本時間を基準とする
- ガント表示は通常の就業時間である08:00〜18:00を対象とする

### 10.4 保守性

- TypeScriptの型を維持する
- 表示コンポーネントとデータ読込処理を分離する
- 既存ロジックを変更するPRでは影響範囲を明記する
- Excelバイナリは通常のPRへ含めない

## 11. v1.0の完成範囲

v1.0では、次を完成範囲とします。

- 閲覧者・管理者の認証分離
- Excelアップロードと入力検証
- 日本時間の日付ナビゲーション
- 職員名・プログラム名・ステータスのフィルター
- 社員別カード表示
- 予定なし表示
- 08:00〜18:00のガント表示
- プログラム別カラー
- 印刷用表示
- Excel本体を公開しないデータ配信

これらが安定して動作する状態を、Program Assign v1.0とします。

## 12. v1.0で行わないこと

次の機能は、Program Assign v1.0には含めません。

- アプリ画面内での予定作成・編集・削除
- ドラッグ＆ドロップによる時間変更
- Program Managerの進捗・期限・リスク管理
- Program Managerとのリポジトリ統合
- チャットや承認ワークフロー
- 個人別の詳細な権限管理
- Excelを廃止したデータベース完全移行

## 13. 今後の拡張候補

優先順位は、実際の利用状況を確認して決定します。

### 優先的に検討する候補

- 予定カードまたはガントからの詳細表示
- 印刷・PDF出力の安定化
- スマートフォン表示の追加調整
- フリーワード検索
- 担当重複の警告
- 現在時刻の担当強調
- 変更履歴と変更通知

### 中長期の候補

- 週間・月間表示
- 職員別ログイン
- Program別表示
- OneDriveまたはGoogle Sheetsとの連携
- Program Managerからの予定データ連携
- OneStop AI Platformからの起動

他アプリとの連携はAPI等で行い、Program Assignの閲覧中心という役割は維持します。

## 14. 機能追加の判断基準

新しい機能は、次の順で判断します。

1. 当日の担当確認を速くするか
2. Excel更新・共有の手間を減らすか
3. 誤認や担当漏れを防ぐか
4. Program Assignの閲覧中心という役割を壊さないか
5. 小さなPRで安全に追加できるか

上記に合わない機能は、Program Managerまたは別アプリで実現することを検討します。

## 15. 変更管理

本書の内容に影響する変更を行う場合は、コード変更と同じPRまたは先行する文書PRで本書を更新します。

バージョンの考え方：

- `v1.x`：基本目的を変えない小規模な機能・運用変更
- `v2.0`：正本、編集方針、アプリの役割など基本設計を変更する場合
